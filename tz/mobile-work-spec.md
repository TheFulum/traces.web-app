## 1. Порядок внедрения и зависимости

Рекомендуемый порядок (граф зависимостей):

```
A (модель Place + i18n полей)
    └── B (каталог), C (главная из config/home), частично H (имена мест)
D (trip planner)
    └── E (инфраструктура рядом на экране маршрута)
F (аккаунт)
    └── синхронизация trips после D; отзывы — после согласования с §14 на web
G, H, I, J — параллельно после A по возможности
```

- **B и C** можно делать параллельно после **A**.
- **E** имеет смысл только когда есть черновой **D** (список точек + карта маршрута).
- **F** можно начинать с анонимного guest-хранилища (избранное/история локально), затем добавить Firebase Auth и merge — см. [`plan.md` §16.5](plan.md).

## 2. Эпик A — Модель `Place` и отображение данных

### Цель

Привести клиентский объект места в соответствие с тем, что уже отдаёт Firestore для сайта:结构化 дата, флаги фильтров, локализованные поля.

### Что сделать

1. Расширить `com.app.traces.model.Place` (и при необходимости ввести вложенные типы / `Map` для гибкости) полями:
   - `createdOn` (объект с `year`, `month`, `day`, `precision`, `sortKey`, опц. `display`) — см. [`plan.md` §3.2.1](plan.md).
   - `hasPhotos`, `has3D` (boolean).
   - Структура **`i18n`** как на сайте (`ru` / `en` / др.) для подстановки имени, описания, адреса, автора — сверить фактические ключи с `js/shared/i18n.js` (`pickI18n`).
   - При необходимости `updatedAt` (Firestore Timestamp) для сортировки «по обновлению».

2. В UI списка и детали:
   - Для заголовка и текстов использовать **локализованное** значение при наличии `i18n`, иначе fallback на корневые `name`, `description`, и т.д. (как `pickI18n` на web).
   - Дату показывать через **`createdOn`** (форматирование по `precision`), при отсутствии — fallback на строковое `openingDate`.

3. Убедиться, что десериализация Firestore не ломается на `null`/отсутствующих полях.

### Как сделать

- Прочитать эталон форматирования даты: `js/shared/placeDate.js` (или аналог в проекте).
- На Android: отдельный утилитный класс `PlaceDisplayKt.java` не нужен если на Java — методы в `Place` или `PlaceFormatting`.
- После изменения модели прогнать загрузку списка и детали на реальных документах из Firestore.

### Критерии приёмки

- Карточка и деталь показывают корректное имя/описание для выбранного языка приложения (после эпика H — переключаемый; до него можно зафиксировать `ru`).
- Фильтры «есть фото / есть 3D» на данных из БД работают без пересканирования массива фото на клиенте (использовать `hasPhotos` / `has3D`, с fallback как в web `computeView` в `js/pages/catalog.js`: `(photos non-empty) || hasPhotos`, то же для 3D).

## 3. Эпик B — Каталог: фильтры и сортировки как на сайте

### Цель

Повторить семантику каталога сайта [`pages/catalog.html`](../../pages/catalog.html) + [`js/pages/catalog.js`](../../js/pages/catalog.js).

### Что сделать

Реализовать состояние фильтров и сортировки:

| Параметр | Поведение (как на web) |
|---|---|
| Поиск `q` | Подстрочное совпадение по имени, адресу, автору, тегам (после A — через `pickI18n` для имени/адреса/автора) |
| `has3d` | true если есть модель: `modelUrl \|\| sketchfabUrl \|\| has3D` |
| `hasphotos` | true если есть фото: `(photos non-empty) \|\| hasPhotos` |
| `city` | Совпадение города из `location.address` — та же функция извлечения города, что на web (`extractCity`), перенести алгоритм 1:1 или вынести в общую спецификацию строки адреса |
| `tags` | **И пересечение:** все выбранные теги должны быть у места |
| `dateFrom` / `dateTo` | Фильтр по **году** из даты создания объекта (`getYearForFilter` — как в `catalog.js`) |
| Сортировка | Варианты: `updated_desc/asc`, `created_desc/asc`, `name_asc/desc`; для даты создания использовать `createdOn.sortKey` с теми же правилами для «нет даты» (в конце списка при asc/desc как на сайте) |

### Как сделать

1. **Вариант 1 (паритет с текущим web):** один раз загрузить все места (`getPlaces` аналог), в памяти применить `computeView` / `sortPlaces` — логику можно почти дословно перенести на Java.
2. **Вариант 2 (масштабирование):** позже — пагинация + серверные запросы; тогда явно зафиксировать отличие от web и добавить индексы Firestore. На первом этапе достаточно варианта 1 при сопоставимом объёме данных.

UI:

- Отдельный экран «Каталог» **или** нижний sheet / expandable блок фильтров на `HomeFragment`.
- Chips для активных фильтров, reset.

Сохранение состояния:

- Локально: `SavedStateHandle` + опционально сохранение в SharedPreferences для следующего запуска.
- Deep link опционально в следующей итерации (на сайте состояние в URL).

### Критерии приёмки

- Для одной и той же выборки Firestore результат фильтрации и порядок элементов совпадают с сайтом при тех же параметрах (ручное сравнение на 5–10 кейсах).

## 4. Эпик C — Главная screen и `config/home`

### Цель

Экран после входа или вкладка «Главная» отражает маркетинговый контент из Firestore документа **`config/home`**, как [`js/pages/index.js`](../../js/pages/index.js) (`applyHomeConfig`).

### Что сделать

1. Загрузка `Firestore.collection("config").document("home")`.
2. Поля для поддержки (минимум паритет с web):
   - `heroImage` — фон верхнего блока (Glide / Coil если перейдёте на Kotlin).
   - `heroEyebrow`, `heroTitle`, `heroSubtitle`, `introText` — локализованные объекты `{ ru: "...", en: "..." }` или строки; функция `pickLocalized` как на сайте.
   - `collections` — массив подборок с полями для карточки (title, description, cover, **filter** — query-string для перехода в каталог с предзаполненными фильтрами).
   - `teasers.mapImage`, `teasers.guideImage` — опционально для блоков-тизеров.
3. Fallback: если документа нет — строки из `strings.xml` / дефолты как в `defaults` в `index.js`.
4. Навигация: тап по коллекции открывает каталог с соответствующим фильтром (парсинг `filter` как на web — см. `defaultCollections()` и `collectionCard` ссылки).

### Как сделать

- Изменить разметку `fragment_home.xml`: сверху hero (ConstraintLayout + ImageView фона), ниже RecyclerView для коллекций, затем опционально блок «избранное с сайта» / быстрые ссылки — по приоритету.
- Отделить данные: `HomeRepository` / `HomeViewModel`, не смешивать с списком всех мест до необходимости.

### Критерии приёмки

- При наличии валидного `config/home` визуально и по текстам главная соответствует задумке сайта (контент совпадает с тем же документом в Firebase Console).

## 5. Эпик D — Trip planner (маршрут поездки)

### Цель

Реализовать сценарий [`plan.md` §16](plan.md): модель поездки, список точек, старт/финиш, оптимизация порядка, линия на карте, сохранение guest и синхронизация после входа.

### Что сделать

1. **Модель `Trip`** (локально + позже Firestore под пользователя): см. §16.2 в `plan.md` — `title`, `start`, `end`, `waypoints` (`placeId[]`), опц. `days`.
2. **UI:**
   - Экран «Мой маршрут»: список с drag-and-drop (`ItemTouchHelper` или библиотека).
   - Кнопки: Оптимизировать (режимы fast / api если решите подключать ORS), Построить, Поделиться, Открыть в навигаторе.
   - На карточке места и в детали: toggle **«В маршрут»** + badge количества точек.
3. **Карта:** слой polyline по точкам маршрута (те же координаты, что у маркеров мест).
4. **Хранение guest:** Room или JSON в SharedPreferences + репозиторий; после эпика F — merge с облаком.

### Как сделать

- Алгоритм **nearest-neighbor + 2-opt** для `optimize=fast` — портировать с web или из §16.3 описание.
- Для `optimize=api` — отдельный HTTP клиент к OpenRouteService / аналог; ключи только на своём backend или ограниченный ключ с квотами (не светить в публичном репозитории без restriction).
- Шаринг: единый формат с сайтом если есть encoded route в URL — свериться с [`pages/route.html`](../../pages/route.html) и связанными JS-модулями; если формат ещё меняется — зафиксировать версию в этом документе.

### Критерии приёмки

- Пользователь добавляет ≥2 места, оптимизирует порядок, видит линию на карте, может открыть внешнюю навигацию.
- После перезапуска приложения черновик маршрута сохранён (guest).

## 6. Эпик E — Инфраструктура рядом (отели / еда)

### Цель

См. [`plan.md` §17](plan.md): POI из OSM (Overpass) около точки или вдоль маршрута — **без бронирований**.

### Что сделать

1. После выбора «финиш в отеле» или точки на карте — запрос Overpass с теми же фильтрами типов (`tourism=hotel|hostel|apartment`, `amenity=restaurant|cafe|fast_food`).
2. UI: список / bottom sheet с названием и расстоянием; тап открывает точку на карте или внешние карты.

### Как сделать

- Использовать тот же подход к запросам и парсингу, что утверждён на web (перенести endpoint builder и обработку ошибок).
- Вынести HTTP в отдельный сервис с таймаутами и отменой при уничтожении экрана.

### Критерии приёмки

- Для тестовой локальности возвращается осмысленный список POI; приложение не падает при пустом ответе / rate limit.

## 7. Эпик F — Аккаунт и личный кабинет (без админки)

### Цель

Паритет с пользовательским web [`plan.md` §14–§15](plan.md): вход, профиль, избранное, история, маршруты, мои отзывы, уведомления — **без** любых функций админки.

### Что сделать

1. Firebase Authentication (email/password или те же провайдеры, что на сайте — зафиксировать один набор).
2. Экраны / разделы:
   - Профиль (имя, email read-only или смена по flow Firebase).
   - Избранное (`placeId[]`).
   - История просмотренных мест (локально + синк опционально).
   - Сохранённые маршруты (из эпика D).
   - Мои отзывы — после того как web зафиксирует модель отзыва с `uid` (§14).
   - Уведомления — если появится коллекция под пользователя на backend.
3. **Merge guest → user:** см. §16.5 в `plan.md`.

### Как сделать

- Структура Firestore для пользовательских данных должна быть **согласована с web** (`users/{uid}/…` или общая схема из репозитория сайта).
- Навигация: пункт «Профиль» в Toolbar или отдельная вкладка (решение UX — не перегружать bottom nav пятью пунктами).

### Критерии приёмки

- Вход и выход работают; данные избранного не теряются при первом логине (merge).

## 8. Эпик G — CMS-страницы (`page.html`)

### Цель

Дать доступ к контентным страницам проекта (не путать с «местами»).

### Варианты реализации

1. **Минимальный:** список документов из коллекции страниц (как на сайте) → открытие в **WebView** по публичному URL страницы на вашем домене + slug.

### Критерии приёмки

Сверка источника данных — с [`pages/page.html`](../../pages/page.html) и JS-модулем загрузки страниц на сайте.

## 9. Эпик H — Локализация

### Цель

Соответствие [`plan.md` §8](plan.md): языки интерфейса как минимум те же, что у сайта.

### Что сделать

1. Вынести строки в `values`, `values-en`, при необходимости `values-be` и др.
2. Настройки приложения: `SharedPreferences` для выбранного языка + `applyOverrideConfiguration` / `AppCompatDelegate.setApplicationLocales` (Android 13+) или attachBaseContext для старых версий.

### Критерии приёмки

- Переключение языка меняет UI без перезапуска процесса (желательно) или с явным перезапуском Activity документированным в README приложения.

## 10. Эпик I — Чат-гид

### Цель

Паритет с [`plan.md` §18](plan.md): единый контракт с Cloudflare Worker (`POST /api/chat` или актуальный URL), ключей OpenRouter в клиенте нет.

### Что сделать

1. Сверить `ChatRepository` с текущим телом запроса/ответа на сайте [`pages/chat.html`](../../pages/chat.html) + соответствующий JS.
2. Добавить действия из §18.3: если ответ содержит предложенный маршрут (`placeId[]`) — кнопка **«Сохранить как маршрут»** создаёт/дополняет `Trip` (эпик D).
3. Rate limiting UX уже частично есть — выровнять сообщения с сайтом.

### Критерии приёмки

- При том же промпте поведение чата функционально совпадает с web (в пределах модели).

## 11. Эпик J — Полировка UX/UI и качество

### Цель

Выровнять визуальный язык с сайтом ([`css/style.css`](../../css/style.css): палитра, контрасты) и закрыть мелочи.

### Что сделать

1. Цветовая схема `colors.xml` / темы Material — приблизить к warm editorial сайта.
2. Убрать англоязычные остатки в UI (например строки thank you в feedback — см. текущие `strings.xml`).

### Критерии приёмки

- Чек-лист ручного прогона на двух языках и на TalkBack для основных потоков.
