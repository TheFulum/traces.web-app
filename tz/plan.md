## 1. Видение, цели и аудитория

### 1.1 Что это за проект

«Следы прошлого» перестают быть «онлайн-справочником с карточками». Новая роль — **гибрид путеводителя и культурной вики**: пользователь приходит не «прочитать про памятник», а **спланировать поездку по памятным местам** или **погрузиться в эпоху** через истории, даты, авторов и 3D.

Метафора: «Wikipedia + Google Maps + цифровой архив», ориентированный на белорусское наследие (мемориалы, скульптуры, архитектура), но расширяемый.

### 1.2 Ключевые ценностные обещания (для пользователя)

1. **«Я могу собрать маршрут».** Каталог с фильтрами + карта, чтобы за 5 минут отобрать места «куда поехать в выходные».
2. **«Я узнаю историю, которой не найду в одной статье».** Длинные описания, авторство, 3D-модель, фото-галерея, отзывы.
3. **«Я вижу место как объект истории, а не точку на карте».** Дата создания, скульптор/архитектор, эпоха, теги.
4. **«Это красиво и приятно листать».** Современные плавные анимации/переходы, сильный визуал главной.

### 1.3 Аудитория и сценарии

| Сегмент | Сценарий | Что нужно от продукта |
|---|---|---|
| **Турист / житель РБ** | Готовится к поездке на выходные | Главная-обзор, каталог с фильтрами, карта, share, маршрут |
| **Школьник / студент** | Готовит реферат / урок | Подробная страница места, дата, автор, ссылки, цитируемость |
| **Краевед / преподаватель** | Ищет малоизвестные памятники по теме | Фильтры по тегам, по эпохе (дата создания), JSON API |
| **Разработчик / исследователь** | Хочет данные программно | Публичный read-only JSON API (раздел [9](#9-публичный-json-api)) |
| **Админ (владелец проекта)** | Поддерживает контент | Удобная админка с переводами и календарём дат |

### 1.4 Не-цели (out of scope для этой итерации)

- Полноценная социальная сеть (профили пользователей, лайки, подписки).
- Платные подписки / монетизация.
- Бронирование / билеты / события.
- Пользовательский UGC сверх отзывов (загрузка фото юзерами).
- Нативное приложение и его функционал — вне этого документа (`tz/mobile-work-spec.md`). Админка только в браузере (см. §10).

### 1.5 Метрики успеха (как поймём, что получилось)

- Среднее время на сайте `> 2 мин` (сейчас оценочно ~30 c).
- Доля сессий, в которых пользователь открывает `>= 3` страницы мест `> 35%`.
- Использование фильтров каталога — `>= 25%` сессий с каталога.
- Lighthouse Performance (mobile) `>= 85`, Accessibility `>= 95`.

## 2. Архитектура и стек

### 2.1 Что есть сейчас (фиксируем)

- **Frontend:** статические HTML страницы + ES-modules (vanilla JS, без сборщика).
- **Стиль:** один `css/style.css` (warm editorial — Playfair Display + Source Sans 3).
- **БД:** Firebase Firestore (`places`, отзывы, feedback).
- **Файлы:** Cloudinary (фото и `.glb`-модели).
- **Карты:** Leaflet + OSM tiles.
- **3D:** Google `model-viewer` для GLB, Sketchfab embed для внешних моделей.
- **Auth админки:** Firebase Auth.
- **Хостинг:** (предположительно Firebase Hosting / GitHub Pages — уточнить).

Принимаем как данность: **не переписываем на React/Vue**. Vanilla + ES-modules сохраняем — это сильно дешевле в поддержке и достаточно для объёма проекта.

### 2.2 Что добавляется

| Назначение | Библиотека / технология | Размер | Альтернатива |
|---|---|---|---|
| Smooth scrolling | **Lenis** (`@studio-freight/lenis`) | ~5 KB gzip | native CSS `scroll-behavior` |
| Scroll-анимации | **GSAP + ScrollTrigger** ИЛИ нативный `IntersectionObserver` | 30 KB / 0 KB | см. раздел 7 |
| Page transitions | **View Transitions API** (нативно) + fallback `barba.js` | 0 / 8 KB | свой роутер |
| i18n | свой `js/i18n.js` (словари + `data-i18n`) | <1 KB | i18next (overkill) |
| Публичный API | **Firebase Cloud Functions** (Node.js, HTTPS trigger) | — | static JSON через CI |
| Календарь дат в админке | **flatpickr** (с локалью) | ~20 KB | `<input type="date">` (не покрывает «только год») |

Подключаем по CDN (как уже принято в проекте: `https://www.gstatic.com/firebasejs/...`, `unpkg`, `ajax.googleapis`). Без npm/bundler.

**Фактические пути в репозитории (на май 2026):** большинство пользовательских страниц лежит в каталоге **`pages/`** (`pages/catalog.html`, `pages/place.html`, `pages/map.html`, `pages/route.html`, `pages/chat.html`, `pages/feedback.html`, `pages/page.html`, `pages/profile/…`, `pages/auth.html`). ### 2.4 Принципы архитектуры кода

- **Один JS-модуль = одна страница**, общая логика — в `js/utils.js`, `js/places.js`, `js/i18n.js`.
- **Никаких глобальных state-менеджеров** — состояние страницы живёт в её модуле.
- **URL — источник истины** для фильтров каталога и языка (`?lang=`, `?has3d=true`, `?sort=date_desc`). Это даёт shareable links и работающий «назад».
- **Все обращения к Firestore через тонкий слой** (`js/places.js`) — упрощает миграцию модели данных и потенциальный переход на Cloud Functions.
- **Никаких изменений правил Firestore «на лету»** — миграционные скрипты в `tz/scripts/` (см. раздел 3.5).

### 2.5 Область документа и нативное приложение

**`plan.md` описывает только веб-сайт:** статические страницы в репозитории (`index.html`, `pages/*.html`), общие стили/скрипты, работу с Firebase со стороны браузера и **веб-админку** (`admin/`).

Модель данных Firestore в §3 остаётся **общим контрактом хранилища** для всех клиентов продукта, но реализация и ТЗ **нативного Android-приложения** ведутся отдельно в **`tz/mobile-work-spec.md`**.

## 3. Модель данных и миграция

### 3.1 Текущая схема `places/{id}` (что есть)

```js
{
  name:           string,
  description:    string,
  openingDate:    string,            // ⚠️ свободная строка — нельзя сортировать
  openingAddress: string,
  author:         string,
  tags:           string[],
  location: { lat: number, lng: number, address: string },
  photos:         string[],          // Cloudinary URLs
  modelUrl:       string | null,
  sketchfabUrl:   string | null,
  modelType:      'file' | 'sketchfab' | null,
  createdAt:      Timestamp,
  updatedAt:      Timestamp
}
```

### 3.2 Что меняем и зачем

#### 3.2.1 Дата создания скульптуры/здания (`createdOn`)

**Проблема.** `openingDate` сейчас строка вида «9 мая 1965 г.» — её нельзя сортировать, нельзя фильтровать по диапазону, нельзя нормально показать на разных языках.

**Требование пользователя.**
- Дата может быть **с разной точностью**: только год / год+месяц / год+месяц+день.
- В админке — ручной ввод `xx.xx.xxxx` ИЛИ календарь.
- Сортировка по этой дате asc/desc в каталоге.

**Решение.** Хранить **структурированный объект** + **числовой sortKey** для индекса/сортировки в Firestore.

```js
createdOn: {
  year:      1965,                  // обязателен
  month:     5,                     // 1..12 или null
  day:       9,                     // 1..31 или null (только если есть month)
  precision: 'day' | 'month' | 'year',
  sortKey:   19650509,              // YYYYMMDD; missing parts = 00
  display:   '9 мая 1965 г.'        // опц., ручная подпись если нужно
}
```

Почему так:
- **`sortKey: number`** — в Firestore это самый дешёвый и удобный способ сортировки (`orderBy('createdOn.sortKey')`). Один числовой индекс.
- Missing parts = `00` сохраняет хронологический порядок: «1965» (`19650000`) идёт **раньше** «9 мая 1965» (`19650509`) — это компромисс, см. ниже.
- **Структура года/месяца/дня** — нужна форматтеру, чтобы рендерить `«1965 г.»`, `«май 1965»`, `«9 мая 1965»` на любом из 3 языков.
- **`precision`** — явная отметка точности (даже если `day === null`, можно отличить «не знаем день» от «забыли заполнить»).
- **`display`** (опционально) — если админ хочет уникальную подпись («лето 1944», «начало XX в.»), её показываем вместо автоформата. На сортировку не влияет, sortKey всё равно есть.

> **Альтернатива, которая хуже:** хранить как `Firestore Timestamp`. Минус: нельзя выразить «только год» — Timestamp всегда полный. Это потеряет точность данных. Не используем.

#### 3.2.2 Новые булевы флаги для фильтров (денормализация)

Фильтры «есть фото» и «есть 3D» можно вычислять на лету из `photos.length > 0` и `modelType !== null`. Но в Firestore нельзя `where('photos.length', '>', 0)` — массивы так не индексируются. Поэтому:

```js
hasPhotos: boolean,   // photos && photos.length > 0
has3D:     boolean    // !!modelUrl || !!sketchfabUrl
```

Эти флаги пересчитываются **на каждом сохранении из админки** — функция `sanitize()` в `js/places.js`. Это денормализация для скорости запросов.

#### 3.2.3 Мультиязычные поля

```js
i18n: {
  ru: { name: '...', description: '...', author: '...', address: '...' },
  en: { name: '...', description: '...', author: '...', address: '...' }
}
// Старые top-level name/description/author/address становятся "ru" по умолчанию
// и фолбэком, если перевода нет.
```

**Правила:**
- `i18n.ru` — обязателен (русский — базовый язык).
- `i18n.en` — опционален. Если перевода нет, фронт показывает русский с пометкой «без перевода» (только для админ-режима) или просто фолбэк (в публичном UI).
- Топ-уровневые `name`, `description` (legacy) **остаются**, синхронизируются с `i18n.ru` для обратной совместимости с поиском/каталогом до полной миграции.

#### 3.2.4 Полные изменения схемы

```js
// places/{id} — после миграции
{
  // legacy fields (синхронизируются с i18n.ru)
  name, description, author,

  // address как структура
  location: {
    lat, lng,
    address: string                  // legacy = ru
  },
  openingAddress: string,            // legacy = ru

  // новая дата
  createdOn: { year, month, day, precision, sortKey, display? } | null,

  // legacy строка-дата — оставляем на 1 спринт ради совместимости, потом удаляем
  openingDate: string,               // DEPRECATED после миграции

  tags: string[],

  photos: string[],
  modelUrl, sketchfabUrl, modelType,

  // флаги для фильтров
  hasPhotos: boolean,
  has3D:     boolean,

  // переводы
  i18n: {
    ru: { name, description, author, address, openingAddress },
    en?: { ... }
  },

  // системные
  createdAt: Timestamp,
  updatedAt: Timestamp
}
```

### 3.3 Индексы Firestore

Добавить composite-индексы (через консоль или `firestore.indexes.json`):

| Поля | Назначение |
|---|---|
| `has3D ASC, name ASC` | фильтр «только с 3D», сортировка по имени |
| `hasPhotos ASC, name ASC` | фильтр «только с фото» |
| `has3D ASC, createdOn.sortKey DESC` | фильтр + сортировка по дате создания |
| `updatedAt DESC` | сортировка «по дате обновления» |
| `createdOn.sortKey DESC` | сортировка «по дате создания» |

> При запуске — Firestore сам предложит создать недостающий индекс (ссылка в консоли в логе ошибки). Не пугаемся, кликаем — он создаётся.

### 3.4 Стратегия миграции

> Все изменения **аддитивны** — старые поля не удаляются до окончательной проверки.

**Шаг 1.** Мерж кода с новой схемой. Функция `sanitize()` в `js/places.js`:
- читает входные `createdOn` (если есть) → пересчитывает `sortKey`;
- проставляет `hasPhotos`, `has3D`;
- зеркалит `i18n.ru.*` → `name/description/...` (legacy).

**Шаг 2.** Скрипт `tz/scripts/migrate-001-dates-and-i18n.mjs`:
1. читает все `places`;
2. для каждого — пытается распарсить `openingDate` (паттерны: `"YYYY"`, `"DD.MM.YYYY"`, `"D месяц YYYY г."`);
3. удалось → пишет `createdOn`, **не трогает** `openingDate` (на случай отката);
4. не удалось → пропускает, выводит в консоль для ручной правки;
5. проставляет `hasPhotos`, `has3D`, копирует существующие `name/description/...` в `i18n.ru`.

Запускается локально через Node + Firebase Admin SDK. Service-account положить в `tz/scripts/.env` (уже в `.gitignore` или добавить).

**Шаг 3.** Ручная проверка через админку 5–10 записей.

**Шаг 4.** Через ~2 недели после стабильной работы — отдельный скрипт удаляет `openingDate` (legacy).

### 4.3 HERO

- **Высота:** `min(100svh, 800px)` — на десктопе fullscreen, на мобиле — full small viewport (учитывает адресную строку, использует `svh`).
- **Фон/эффект:** WebGL hero-сцена как ключевой первый акцент на главной (один эффект в первом экране; остальной сайт без WebGL).
- **Поверх фотографии:**
  - eyebrow `Путеводитель по памятным местам`
  - `<h1>` крупный (`clamp(2.5rem, 8vw, 6rem)`), `font-display`, италик слова «прошлого».
  - подзаголовок 1–2 строки.
  - **Две кнопки CTA:**
    - **[Открыть каталог]** — primary, на `catalog.html`.
    - **[На карту]** — outline, на `map.html`.
  - В правом нижнем углу: **скролл-индикатор** (мышка/стрелка вниз) с лёгкой анимацией.
- **Анимация при загрузке:** последовательное появление eyebrow → h1 (по словам/буквам, GSAP SplitText или CSS-задержки) → подзаголовок → кнопки. Длительность ~1 c.
- **На скролле** hero уезжает с лёгким параллаксом (фон движется чуть медленнее текста, `transform: translateY(scrollY * 0.3)`).

### 4.4 Intro Text Block

- Заголовок секции (eyebrow + H2, как в текущем дизайне).
- Текст: 2 коротких абзаца. Пример:
  > *«Здесь живут памятники, мемориалы, скульптуры и места событий, изменивших мир. Каждая карточка — это история: автор, эпоха, фотографии, иногда — 3D-модель, по которой можно «обойти» памятник прямо в браузере. Соберите свой маршрут по городу, изучите архив или просто пройдитесь по эпохам.»*
- Под ним — **3 счётчика** в стиле hero-stats: «N мест», «N городов», «N эпох» (диапазон лет от min до max `createdOn.year`). Анимация числа (count-up) при появлении в viewport.
- Под счётчиками — **3 ссылки-указателя**: «Каталог →», «Карта →», «Гид →» (большие текстовые ссылки с подчёркиванием при ховере).

### 4.5 Collections (тематические подборки)

- 3–6 карточек, расположены в горизонтальном grid (на десктопе) или скролле (на мобиле).
- Каждая подборка = заранее подготовленный фильтр в каталоге, например:
  - «Памяти ВОВ» → `catalog.html?tags=ВОВ,Мемориал`
  - «Скульптуры Минска» → `catalog.html?tags=Скульптура&city=Минск`
  - «До 1900 года» → `catalog.html?dateTo=1900&sort=date_asc`
  - «С 3D-моделью» → `catalog.html?has3d=true`
- Карточка подборки: фоновое фото-коллаж/выбранная фотография + большой заголовок + краткое описание (1 строка) + «N мест» + arrow-CTA.
- Конфиг подборок — отдельный документ Firestore `config/collections` (массив объектов), редактируется в админке (раздел [10](#10-админка--изменения)) или вручную через консоль.

### 4.6 Highlighted Places

- Сетка 3×2 или 4×2 на десктопе (из карточек, идентичных каталожным).
- Источник: `places` где `featured == true`, сортировка по `featuredOrder` или `updatedAt`. Если нет featured — фолбэк на 6 последних обновлённых.
- Под секцией — крупный CTA `[Все места →]`.

### 5.2 Фильтры

| Фильтр | Тип | Значения | URL-параметр |
|---|---|---|---|
| 3D-модель | toggle | `true` / любой | `has3d=true` |
| Фотографии | toggle | `true` / любой | `hasphotos=true` |
| Теги | мульти-select | список тегов из всех мест | `tags=Скульптура,ВОВ` |
| Город | select | уникальные города | `city=Минск` |
| Диапазон даты создания | from/to | year (number) | `dateFrom=1900&dateTo=1990` |

**UI.** Отдельная панель/модалка (на мобиле) или dropdown-чекбоксы (на десктопе). Сверху над сеткой — **chips** активных фильтров с крестиком для удаления. Кнопка «Сбросить всё».

### 5.3 Сортировки

| Опция | Поле Firestore | Дефолт |
|---|---|---|
| По дате обновления (новые → старые) | `updatedAt DESC` | ✅ default |
| По дате обновления (старые → новые) | `updatedAt ASC` |  |
| По дате создания (старые → новые) | `createdOn.sortKey ASC` |  |
| По дате создания (новые → старые) | `createdOn.sortKey DESC` |  |
| По названию (А→Я) | `name ASC` |  |
| По названию (Я→А) | `name DESC` |  |

URL: `?sort=updated_desc | updated_asc | created_asc | created_desc | name_asc | name_desc`.

> **Замечание про sortKey.** Места с `createdOn = null` при сортировке по дате создания должны идти **в конец** (для обоих направлений). Решение: считать таким местам `sortKey = -1` для ASC (но тогда они впереди) или вторичная сортировка через клиентский post-process. Реализовать через клиентский `[...].filter(...).sort(...)` после `getDocs` для выбранного режима — простой и предсказуемый.

### 5.4 Поиск

- Локальный поиск (как сейчас) — клиент скачивает все места, фильтрует в памяти.
- Поля поиска: `name`, `address`, `tags`, `author` (текущего активного языка + русский как fallback).
- Дебаунс 200 мс.
- Подсветка совпадений в карточке (опц., можно отложить).

### 5.5 URL-state

**Все фильтры и сортировка — в query-string.** Это даёт:
- shareable links (ровно то, что просили в подборках главной),
- работающую кнопку «назад»,
- возможность сделать «избранные пресеты».

Реализация — `js/catalog.js`:
- читает `URLSearchParams` при загрузке;
- обновляет URL через `history.replaceState` при каждой смене фильтра;
- никаких hash-роутингов.

### 5.6 Пагинация

При большом количестве мест (`> 50`):
- **Infinite scroll** через `IntersectionObserver` на «sentinel» внизу.
- Размер страницы: 24.
- Курсор Firestore: `startAfter(lastDocSnapshot)`.

Если поиск активен — поиск работает по уже загруженным; при текущих объёмах (десятки/сотни мест) можно грузить всё разом.

### 5.7 Пустые состояния

| Случай | Сообщение |
|---|---|
| Каталог пуст (БД пустая) | «Скоро здесь появятся первые места» + ссылка на feedback |
| Найдено 0 по фильтру | «Ничего не нашлось. Попробуйте сбросить фильтры.» + кнопка [Сбросить] |
| Ошибка загрузки | «Не удалось загрузить места. Проверьте подключение.» |

### 5.8 Анимация появления карточек

При первой загрузке и при подгрузке — **stagger reveal** (см. раздел [7](#7-анимации-и-переходы-lusion-стиль)):
- начальное состояние: `opacity: 0, translateY(24px)`;
- появление при пересечении viewport;
- задержка между соседними картами 60–100 мс;
- `prefers-reduced-motion` — мгновенно.

### 6.4 Share — формат «как у YouTube»

Сейчас `js/place.js:298–325` собирает большой многострочный текст со всем описанием и т.п. Заменяем на короткий формат:

```
{Название} — Следы прошлого
{URL}
```

Пример:
```
Хатынь — Следы прошлого
https://traces.example/place.html?id=hatyn
```

Логика:
- `navigator.share` (мобила) получает `{ title, text, url }`, где `title = place.name`, `text = "{name} — Следы прошлого"`, `url = href`.
- Fallback (десктоп / нет Web Share API) — копируется в буфер только URL (короче) + toast «Ссылка скопирована». Альтернатива: копировать `"{name} — Следы прошлого\n{url}"` — обсудить.
- **Удалить** дублирование описания/автора/адреса в share — это утяжеляло сообщение.

### 6.5 3D Sketchfab — мобильный режим без UI

**Проблема.** Sketchfab embed на мобиле показывает контролы (логотип, "Open in Sketchfab", меню), пользователь может уйти с сайта.

**Решение.** Добавить в URL embed-параметры, скрывающие UI **только на мобиле** (`window.matchMedia('(max-width: 768px)').matches`):

```
?autostart=1
&ui_controls=0
&ui_infos=0
&ui_inspector=0
&ui_stop=0
&ui_watermark=0
&ui_hint=0
&ui_settings=0
&ui_vr=0
&ui_fullscreen=0
&ui_annotations=0
&ui_loading=0
&ui_help=0
&ui_theme=dark
&autospin=0.2
```

> Полный список параметров — официальная документация Sketchfab Viewer API. На десктопе оставляем дефолтные (с логотипом и контролами Sketchfab — таково условие лицензии бесплатного тарифа; в платных аккаунтах можно скрыть и на десктопе).

**Модификация `js/place.js:225-251` (renderSketchfab):**

```js
const isMobile = window.matchMedia('(max-width: 768px)').matches;
const params = isMobile
  ? '?autostart=1&ui_controls=0&ui_infos=0&ui_inspector=0&ui_stop=0&ui_watermark=0&ui_hint=0&ui_settings=0&ui_vr=0&ui_fullscreen=0&ui_annotations=0&ui_help=0&ui_theme=dark'
  : '?autostart=1&ui_theme=dark';
embedUrl = `https://sketchfab.com/models/${id}/embed${params}`;
```

**Дополнительно для надёжности.**
- На мобиле iframe оборачивается в `<div>` с `pointer-events: auto` (как сейчас), но **поверх** добавляем прозрачный overlay только в зонах вокруг (не в центре), чтобы блокировать случайные тапы по «логотипу Sketchfab», но позволить вращение модели в центре. Это **не реализовываем** в первой итерации — сначала смотрим, достаточно ли URL-параметров.
- `<iframe sandbox="allow-scripts allow-same-origin">` — без `allow-popups`, `allow-top-navigation`, чтобы Sketchfab не мог открыть внешнюю вкладку.

## 7. Анимации и переходы (lusion-стиль)

### 7.1 Что хочется и что **возможно**

Пользователь упомянул [lusion.co](https://lusion.co) как референс. На lusion работает:

| Эффект lusion | Реалистично? | Чем делать |
|---|---|---|
| Smooth inertia scroll | ✅ да | Lenis |
| Scroll-revealed элементы (fade/lift) | ✅ да | GSAP ScrollTrigger или native IntersectionObserver |
| Параллакс блоков и фонов | ✅ да | CSS transform + scroll listener (Lenis даёт значение) |
| Stagger-анимации сеток | ✅ да | GSAP timeline / CSS transition delays |
| Page transitions (мягкая смена страниц) | ✅ да (с оговорками) | View Transitions API + barba.js fallback |
| WebGL hero-сцены, image distortion | ✅ да, точечно | Three.js + glsl shaders (только hero на главной) |
| 3D прокрутка модели по скроллу | ⚠️ дорого | model-viewer + custom JS. Опция на будущее. |
| Сложная морфинг-анимация курсора | ⚠️ перебор | Кастомный курсор GSAP. Опц., не приоритет. |

> **Вывод:** делаем «lusion-feel light + WebGL hero». Один цепляющий WebGL-эффект только в hero на главной (первое впечатление). Остальные разделы и страницы — без WebGL: smooth scroll, плавные scroll-reveal, page transitions, аккуратный параллакс, stagger-сетки.

### 7.2 Использование спаршенного `lusion.co`

Парсер пользователя сохранил структуру и часть JS lusion. **Нельзя использовать как основу** напрямую (правовые причины + код перепутан в build-чанках). Что **можно**:

1. **Открыть в браузере, изучать ритм и тайминги** анимаций — какие задержки, какие easings.
2. Посмотреть, какие CSS-переменные использованы, какие `clip-path`/`transform`-комбинации.
3. **Адаптировать идеи** (не код): тип переходов, вид staggered grid, типографика.

> В коде traces ничего из спаршенного lusion **не копируется 1-в-1**. Только идеи и тайминги.

### 7.3 Smooth scroll (Lenis)

Подключение:
```html
<script src="https://unpkg.com/lenis@1.x/dist/lenis.min.js"></script>
```

Инициализация в `js/scroll.js`:
```js
const lenis = new Lenis({
  duration: 1.2,
  easing: t => Math.min(1, 1.001 - Math.pow(2, -10 * t)),
  smoothWheel: true,
  smoothTouch: false   // на мобиле нативный скролл — экономнее
});
function raf(time) { lenis.raf(time); requestAnimationFrame(raf); }
requestAnimationFrame(raf);
window.lenis = lenis;
```

Применяется глобально на всех страницах сайта (подключаем в каждой странице через `<script type="module" src="js/scroll.js"></script>`).

Отключение по `prefers-reduced-motion`:
```js
if (matchMedia('(prefers-reduced-motion: reduce)').matches) return; // не запускать lenis
```

### 7.4 Scroll-reveal (плавное появление)

Подход — **IntersectionObserver, без GSAP**, чтобы не таскать 30 КБ ради простого fade. (GSAP подключаем только если будут таймлайны посложнее.)

CSS:
```css
.reveal {
  opacity: 0;
  transform: translateY(24px);
  transition: opacity .8s cubic-bezier(.22,1,.36,1),
              transform .8s cubic-bezier(.22,1,.36,1);
  will-change: opacity, transform;
}
.reveal.is-in { opacity: 1; transform: none; }

/* для grid-staggered, отдельные классы или CSS variable */
.reveal--stagger { transition-delay: calc(var(--i, 0) * 80ms); }
```

JS (`js/reveal.js`):
```js
const io = new IntersectionObserver((entries) => {
  entries.forEach(e => {
    if (e.isIntersecting) {
      e.target.classList.add('is-in');
      io.unobserve(e.target); // одноразово
    }
  });
}, { threshold: 0.15, rootMargin: '0px 0px -50px 0px' });

document.querySelectorAll('.reveal').forEach((el, i) => {
  el.style.setProperty('--i', i % 8);
  io.observe(el);
});
```

`prefers-reduced-motion`: убираем класс `reveal` в JS либо добавляем CSS-фолбэк:
```css
@media (prefers-reduced-motion: reduce) {
  .reveal { opacity: 1; transform: none; transition: none; }
}
```

### 7.5 Параллакс блоков

Простой вариант — на каждом блоке-параллаксе атрибут `data-parallax="0.3"`:
```js
lenis.on('scroll', ({ scroll }) => {
  document.querySelectorAll('[data-parallax]').forEach(el => {
    const speed = parseFloat(el.dataset.parallax);
    const offset = (scroll - el.offsetTop) * speed;
    el.style.transform = `translate3d(0, ${offset}px, 0)`;
  });
});
```

Используем экономно: hero-фон, фон map-teaser, может быть один-два блока в Collections.

### 7.6 Page transitions

**Цель:** при клике на ссылку внутри сайта страница не «мигает» — есть мягкий fade/slide.

#### 7.6.1 View Transitions API (предпочтительно)

Поддерживается в Chrome/Edge `>= 111`, Safari TP. Работает только для same-origin переходов.

Подключение:
```js
// js/transitions.js
document.addEventListener('click', e => {
  const a = e.target.closest('a[href]');
  if (!a) return;
  const url = new URL(a.href, location.href);
  if (url.origin !== location.origin) return;
  if (a.target === '_blank') return;
  if (!document.startViewTransition) return; // фолбэк = нативный переход

  e.preventDefault();
  document.startViewTransition(() => {
    location.href = a.href;
  });
});
```

CSS:
```css
::view-transition-old(root) { animation: vt-fade-out .3s ease both; }
::view-transition-new(root) { animation: vt-fade-in  .4s ease both; }
@keyframes vt-fade-out { to { opacity: 0; transform: translateY(-8px); } }
@keyframes vt-fade-in  { from { opacity: 0; transform: translateY(8px); } }
```

Дополнительно: общие элементы (например, кликнутая карточка → hero на странице места) можно связать через `view-transition-name: place-{id}`. На карточке и на h1 страницы места задаём одинаковое имя — браузер сам сделает morph. Это даёт wow-эффект уровня lusion.

#### 7.6.2 Fallback (Safari старее, Firefox)

Подключаем `barba.js`:
```html
<script src="https://unpkg.com/@barba/core"></script>
```

Простейший wrapper: `<div data-barba="wrapper"><main data-barba="container">...</main></div>` на каждой странице. Транзишены — те же fade/slide.

> **Реалистично:** начинаем только с View Transitions API (без barba). Если в Safari совсем плохо — добавляем barba во второй итерации.

### 7.7 Hero-анимации главной

- При первой загрузке — приостановить lenis на 800 мс, проиграть hero-intro (см. 4.3), потом включить scroll.
- SplitText (нативный — простой `string.split(' ').map(w=>'<span>')`) для заголовка по словам, появление с задержкой `i * 60ms`.
- Скролл-индикатор — бесконечная мягкая bounce-анимация.

### 7.8 Производительность

- Использовать `transform/opacity` (compositor-only). Никогда не анимировать `top/left/width/height`.
- `will-change` ставить **только перед** анимацией и снимать после.
- Мобильные оптимизации обязательны и выполняются параллельно с web-реализацией (без отдельной админки).
- Все анимации соблюдают `prefers-reduced-motion: reduce` — отключаются полностью.

### 7.9 Ограничения и что не делаем

- ❌ Нет WebGL за пределами hero на главной.
- ❌ Нет custom-курсора (опц. на будущее).
- ❌ Не делаем 1-в-1 копию lusion — это и невозможно (там WebGL), и не нужно (тема проекта другая — culture/heritage, а не studio-portfolio).

## 8. Мультиязычность

### 8.1 Языки

| Код | Язык | Приоритет |
|---|---|---|
| `ru` | Русский | базовый, дефолт |
| `en` | English | вторая очередь |

`ru` — обязателен; `en` — опционален на уровне записей мест (если перевода нет — фолбэк на `ru`).

### 8.2 Что переводится

| Слой | Способ | Источник |
|---|---|---|
| **UI** (кнопки, заголовки, плейсхолдеры) | словарь в коде | `js/i18n/locales/{ru,en}.json` |
| **Контент мест** (название, описание, автор, адрес) | поля документа | `places/{id}.i18n.{ru,en}` |
| **Главная** (hero, intro, collections) | поля Firestore | `config/home.heroTitle.{ru,en}` и т.п. |
| **Системные тексты** (404, errors) | словарь | `js/i18n/locales/...` |
| **Даты** (форматирование) | утилита | `formatPlaceDate(createdOn, lang)` |
| **Отзывы** | не переводятся | пишутся как написал автор |

### 8.3 Архитектура `js/i18n.js`

```js
// js/i18n.js
const SUPPORTED = ['ru', 'en'];
const DEFAULT_LANG = 'ru';

export function getLang() {
  // приоритет: ?lang=  >  localStorage  >  navigator.language  >  default
  const fromUrl = new URLSearchParams(location.search).get('lang');
  if (SUPPORTED.includes(fromUrl)) return fromUrl;
  const fromLs = localStorage.getItem('lang');
  if (SUPPORTED.includes(fromLs)) return fromLs;
  const nav = (navigator.language || '').slice(0, 2);
  if (SUPPORTED.includes(nav)) return nav;
  return DEFAULT_LANG;
}

export function setLang(lang) {
  if (!SUPPORTED.includes(lang)) return;
  localStorage.setItem('lang', lang);
  const url = new URL(location.href);
  url.searchParams.set('lang', lang);
  location.href = url.toString();   // полная перезагрузка — проще, чем live-перевод
}

export async function loadDict(lang) {
  const res = await fetch(`/js/i18n/locales/${lang}.json`);
  return res.json();
}

export function applyDict(dict) {
  document.documentElement.lang = dict.__lang;
  document.querySelectorAll('[data-i18n]').forEach(el => {
    const key = el.dataset.i18n;
    const val = key.split('.').reduce((o, k) => o?.[k], dict);
    if (val) el.textContent = val;
  });
  document.querySelectorAll('[data-i18n-placeholder]').forEach(el => {
    const key = el.dataset.i18nPlaceholder;
    const val = key.split('.').reduce((o, k) => o?.[k], dict);
    if (val) el.placeholder = val;
  });
}

export function t(key, dict) {
  return key.split('.').reduce((o, k) => o?.[k], dict) || key;
}

export function pickI18n(obj, lang) {
  // для контента из Firestore: obj.i18n?.[lang]?.field || obj.i18n?.ru?.field || obj[field]
  return obj?.i18n?.[lang] || obj?.i18n?.ru || obj;
}
```

## 9. Публичный JSON API

### 9.1 Цель

Дать сторонним разработчикам, исследователям, городским сервисам **read-only** программный доступ к данным мест в формате JSON.

### 9.2 Технология

**Firebase Cloud Functions (HTTPS trigger)** — самый дешёвый и надёжный путь:
- единый аккаунт (тот же Firebase проект `traces-ffd5c`),
- кеш через CDN (Firebase Hosting integration),
- легко добавить rate limit / API-ключи на втором шаге.

> **Альтернативы, которые не выбираем:**
> - Прямой доступ к Firestore из чужого SDK с публичными правилами — неудобно, нет контроля схемы.
> - Статический JSON-снапшот, генерируемый по cron — теряет свежесть.

### 9.3 Endpoints

Все endpoints — `GET`, ответ `application/json`, CORS открыт (`Access-Control-Allow-Origin: *`).

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/v1/places` | Список мест (с фильтрами и пагинацией) |
| `GET` | `/api/v1/places/{id}` | Одно место |
| `GET` | `/api/v1/tags` | Уникальные теги (для UI, опц.) |
| `GET` | `/api/v1/health` | Healthcheck `{status:"ok",version:"v1"}` |

### 9.4 `GET /api/v1/places`

| `hasphotos` | `true \| false` | фильтр |
| `tags` | csv | `tags=Скульптура,ВОВ` (AND или OR — обсудить) |
| `dateFrom` | int (year) | дата создания с |
| `dateTo` | int (year) | дата создания по |
| `sort` | enum (см. 5.3) | сортировка |

### 9.5 Версионирование

- Префикс `/v1/` — обещание стабильного контракта.
- Любые breaking changes — `/v2/` параллельно. Старая версия живёт ≥ 6 мес.
- Не-breaking (новые поля) — без изменения версии.

### 9.6 CORS, ошибки, rate limiting

- `Access-Control-Allow-Origin: *` (read-only API, не страшно).
- Ошибки в формате:
  ```json
  { "error": { "code": "not_found", "message": "Place not found" } }
  ```
- HTTP коды: `200`, `400` (плохие параметры), `404`, `429` (rate limit), `500`.
- **Rate limit (вторая итерация):** `60 requests / minute / IP` — через `firestore`-счётчик или Firebase App Check. На старте — без лимита.
- **Кеширование:** заголовок `Cache-Control: public, max-age=300` (5 мин).

### 9.7 Документация — `api.html`

Отдельная страница на сайте с:
- описанием endpoints (можно даже просто markdown-rendered),
- примерами `curl` / `fetch`,
- ссылкой на репозиторий (если опубликован),
- блоком «зачем это и как нам сообщить о проблеме».

OpenAPI 3.0 spec (`tz/openapi.yaml`) — приветствуется, но не обязателен в первой итерации.

### 9.8 Безопасность

- API только READ. POST/PUT/DELETE — нет, никогда.
- Не возвращаем email/имена, оставленные в feedback или review (приватные данные).
- Service account ключ Firebase — только в env переменных Cloud Functions, никогда в репо.

## 10. Админка — изменения

### 10.1 Что меняем в форме редактирования места

**Текущее поле «Дата открытия» (свободная строка)** заменяем на блок «Дата создания»:

```
┌── Дата создания / открытия ──────────────────┐
│  Точность:  ◉ Год   ○ Год+месяц   ○ Полная   │
│                                                │
│  ┌────────┐  ┌──────────┐  ┌──────────┐       │
│  │  1965  │  │ Май  ▾   │  │   9   ▾  │       │
│  └────────┘  └──────────┘  └──────────┘       │
│  Год         Месяц          День               │
│                                                │
│  Или ввести вручную: ┌────────────────┐       │
│                       │ 09.05.1965     │       │
│                       └────────────────┘       │
│                                                │
│  Или подпись свободным текстом (опц.):        │
│  ┌──────────────────────────────────┐         │
│  │ лето 1944, начало XX в.          │         │
│  └──────────────────────────────────┘         │
│  ↑ если задано — будет показано вместо авто   │
└────────────────────────────────────────────────┘
```

**Поведение:**
1. Радио-группа `precision`: год / год+месяц / полная.
2. Поля `год / месяц / день` — disabled в зависимости от `precision`.
3. Поле «вручную» — парсит `DD.MM.YYYY`, `MM.YYYY`, `YYYY`. При успешном парсинге заполняет селекторы. При невалидном формате — красный бордер + сообщение.
4. Поле «подпись» (`display`) — необязательное, не влияет на сортировку.
5. На сохранении — `sortKey` пересчитывается на клиенте (`sanitize()` в `places.js`).
6. **Используем `flatpickr`** для красивого календаря в режиме «полная дата». В режимах «только год/год+месяц» — простые `<select>`.

### 10.2 Переводы

Под каждым переводимым полем (`name`, `description`, `author`, `address`, `openingAddress`) — **табы языков**:

```
┌───── Название ──────────────────────────┐
│  [RU]  [EN]                              │  ← табы
│  ┌────────────────────────────────────┐  │
│  │ Хатынь                             │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
```

- RU обязателен.
- EN — на табе badge «не заполнено», если поле пустое.
- При сохранении: значения каждого таба пишутся в `i18n.{lang}.{field}`.

### 10.5 3D-модель — без изменений

### 10.6 Управление главной страницей (`/admin/home.html`)

- **robots.txt** разрешает индексацию, кроме `/admin/`.

### 11.3 Доступность (a11y)

- **Контраст** проверить для всех пар (текст vs фон) — WCAG AA как минимум.
- **Фокус-стили:** добавить заметный outline для всех интерактивных элементов (`:focus-visible`).
- **ARIA:** на nav-burger, lightbox, модалках — `aria-label`, `aria-expanded`, `aria-modal`.
- **Клавиатурная навигация:** lightbox (Esc/←/→ — уже есть), модалки админки (Esc), фокус-trap внутри модалок.
- **`prefers-reduced-motion: reduce`** — отключает Lenis, scroll-reveal, параллакс, page transitions (см. раздел 7).
- **Альт-тексты:** все `<img>` имеют осмысленный `alt`. Декоративные иконки — `alt=""` или `aria-hidden="true"`.
- **Семантика:** `<nav>`, `<main>`, `<aside>`, `<article>`, `<section>` используются по назначению.

### 11.4 Производительность анимаций

- На мобиле тестировать на устройствах среднего класса (можно throttle CPU 4× в DevTools).
- Если FPS проседает — уменьшать число одновременных параллакс-блоков, отключать Lenis на touch (как сейчас планируем).
- При появлении карточек grid (`stagger-reveal`) — лимит 8 одновременных в анимации, остальные ждут `IntersectionObserver`.

### 11.5 Аналитика (минимальная)

- **Plausible** или **Umami** (privacy-friendly, без cookies). Cloudflare Web Analytics как самый дешёвый.
- События: открыта главная, клик на коллекцию, использован фильтр, открыто место, нажата «поделиться», нажата «маршрут».

### 11.6 Адаптив — обязательное требование (DoD для web)

Адаптив фиксируется как **must-have**, а не как опциональная полировка.

Базовые брейкпоинты:

- `1024–1279` desktop

- `480–767` mobile

## 16. Маршруты (Trip planner) и оптимизация пути

> Добавлено по результатам обсуждения. Требование относится к **пользовательскому сайту** (не к админке).

### 16.1 Цель

Сделать сайт **инструментом планирования поездки**, а не только библиотекой карточек: пользователь выбирает точки, задаёт старт/финиш (дом/отель), получает оптимальный порядок и маршрут на карте, сохраняет и делится.

### 16.2 Модель маршрута

Маршрут — это сущность `trip` (по сути “план поездки”):

- `title` (опц.)
- `start`: одна из стратегий:
  - `geo` (lat/lng) — “дом” (геолокация) или точка на карте
  - `placeId` (редко, но возможно)
- `end`: стратегия:
  - `lastPoint` (закончить на последней точке)
  - `returnToStart` (вернуться домой)
  - `geo` (например, “в отель”)
- `waypoints`: список `placeId[]` (выбранные места)
- `days` (опц.): разбиение на дни `day[]`, где у каждого дня свой `start/end` (например, ночёвка в отеле)

### 16.3 Оптимизация порядка точек (важно)

Требование: маршрут не должен “прыгать” дом → восток → дом → запад → дом → юг.  
Пользователь выезжает из дома и едет по точкам **в разумном порядке**, с возможностью остановиться/ночевать.

Подход (реалистичный для диплома и бюджета):

- **Start/End aware**: оптимизация всегда учитывает старт (дом) и опциональный финиш (дом/отель/последняя точка).
- **Два режима**:
  1. `optimize=fast` (дефолт): быстрый эвристический порядок (nearest-neighbor + 2-opt) — работает везде и дешево.
  2. `optimize=api` (если доступно по квотам): оптимизация через routing API (например, OpenRouteService Optimization/VRP), тоже с фиксированным start/end.

### 16.4 Маршрут на карте

- Показ polyline на карте (**Leaflet** на страницах сайта).
- Подсчёт суммарного времени/дистанции.
- Экспорт/навигация: “Открыть маршрут в навигаторе” (ссылки на Google / Яндекс / OSM и т.п. — как принято на сайте).

### 16.5 Сохранение и синхронизация (guest → account merge)

Требование: маршрут может собрать **любой** пользователь без аккаунта.

- **Guest mode в браузере:** `localStorage` (или согласованное хранилище для `trip`/избранного до входа).
- **После логина**: **merge** без потери данных:
  - локальные trips/favorites/history вливаются в аккаунт;
  - если в аккаунте уже есть — объединение без дублей.

### 16.6 UI/UX (минимум)

- На карточках/странице места: кнопка **“В маршрут”** (toggle) + быстрый доступ к “Мой маршрут”.
- Отдельная страница **`pages/route.html`** (или эквивалентный URL после деплоя):
  - список точек (drag & drop порядок)
  - выбор start/end
  - кнопки “Оптимизировать”, “Построить”, “Поделиться”
- Состояния: пустой маршрут, ошибка построения, офлайн.

- Показ “где поесть” и “где остановиться” **рядом с выбранной точкой/маршрутом**.
- Фильтры: тип (hotel/restaurant/cafe), радиус, (опц.) рейтинг/ценовой уровень если доступно в источнике.
- Встраивание сторонних сайтов через `<iframe>` **не используем** (часто запрещено CSP/X-Frame-Options, слабый UX).

### 17.2 Источник данных

- Базовый источник: **OpenStreetMap** (например, Overpass API) для POI:
  - `tourism=hotel|hostel|apartment`
  - `amenity=restaurant|cafe|fast_food`

Ограничение: “свободно/занято сейчас” (availability) не гарантируется в бесплатных открытых данных — делаем упор на **поиск инфраструктуры**, а не на бронь.

## 18. Чат-гид: OpenRouter (ключи не в клиенте)

### 18.1 Требование безопасности

- Ключ OpenRouter **никогда** не хранится в браузерном клиенте сайта.
- Доступ к OpenRouter идёт только через server-side proxy.

### 18.2 Proxy через Cloudflare Worker (выбранный вариант)

- OpenRouter API key хранится в **Cloudflare Worker secrets**.
- Страница чата на сайте обращается к Worker endpoint (например, `POST /api/chat`), Worker проксирует в OpenRouter.
- На Worker обязательны ограничения:
  - rate limit (IP + uid если есть)
  - лимиты на размер prompt/ответа
  - allowlist моделей (запрет дорогих/нежелательных)

### 18.3 Чат как инструмент планирования

Чат должен уметь отдавать структурированный результат:

- “подборка/фильтры” (tags/city/has3d/date range)
- “предложенный маршрут” (placeId[])

UI на странице чата: кнопка **«Сохранить как маршрут»** (перенос предложенных `placeId` в сценарий §16).
