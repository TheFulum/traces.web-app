# Мини-ТЗ: обновление Android-клиента до web-паритета

## 1) Цель

Обновить Android-приложение `D:/data/source/Android/traces` до функционального и визуального паритета с текущей web-версией (кроме админки), без полного переписывания проекта с нуля.

## 2) Границы работ

### Входит
- Публичные пользовательские экраны:
  - Home
  - Catalog
  - Place Details
  - Map
  - Chat
  - Feedback
  - Auth / Profile
- Отзывы о местах, привязанные к аккаунту.
- Базовая визуальная полировка UI под текущий стиль web.

### Не входит
- Админка на Android.
- Полная смена стека (Firebase сохраняется).
- Радикальный редизайн или полный переход на Compose в этой итерации.

## 3) Ключевые требования

### 3.1 Данные и совместимость
- Поддержать актуальную модель мест:
  - `createdOn { year, month, day, precision, sortKey, display }`
  - `hasPhotos`, `has3D`
  - `i18n.ru/en` с фолбэком на legacy поля.
- Ввести слой маппинга DTO -> UI model для мягкой миграции legacy данных.

### 3.2 Auth + Profile
- Добавить регистрацию/логин/логаут.
- Добавить экран профиля:
  - изменение имени,
  - смена email/пароля (через re-auth),
  - блок «Мои отзывы».
- Сохранять и восстанавливать сессию пользователя.

### 3.3 Отзывы
- Отправка отзыва только от авторизованного пользователя.
- Привязка отзыва к `uid` + `displayName`.
- В профиле показать список последних отзывов пользователя.

### 3.4 Catalog parity
- Добавить/доработать отдельный каталог с фильтрами:
  - 3D
  - Фото
  - Теги
  - Город
  - Диапазон даты
  - Сортировки (`updated`, `created`, `name`).
- Пагинация и корректные empty/error states.

### 3.5 UI/UX
- Привести отступы, типографику, карточки, состояния кнопок к web-стилю.
- Сохранить нативную мобильную навигацию.
- Обеспечить плавный скролл без деградации жестов.

## 4) Этапы

### Этап A — Контракты данных (обязательный)
1. Обновить модели `Place`, `Review`, профиль пользователя.
2. Добавить мапперы с поддержкой legacy.

### Этап B — Аккаунты и отзывы
1. Auth-экран (вход/регистрация).
2. Profile-экран.
3. Привязка отзывов к аккаунту.

### Этап C — Каталог
1. Доработать фильтры/сортировки.
2. Внедрить пагинацию и стабильные состояния загрузки.

### Этап D — Полировка и стабилизация
1. UI-полировка основных экранов.
2. Smoke-check ключевых сценариев.

## 5) Критерии готовности (DoD)

- Все публичные сценарии web доступны в Android (без админки).
- Отзывы публикуются от аккаунта пользователя.
- Профиль позволяет редактировать данные и видеть свои отзывы.
- Каталог фильтрует/сортирует корректно.
- Нет критичных проблем со скроллом/навигацией на мобильных устройствах.

## 6) Риски и ограничения

- Возможен дрейф схемы между web и Android при параллельной разработке.
- Без выделенного API-слоя часть бизнес-логики останется дублированной.
- В этой итерации приоритет — функциональный паритет, не идеальный pixel-perfect.
